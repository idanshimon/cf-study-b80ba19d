What We're Testing

ContextForge is an Apache-2.0 Python SDK (2,389 LOC) acting as a middleware between an app and an LLM. It ingests a corpus into a SQLite knowledge tree and retrieves a "working set" per query via an in-memory BM25 keyword index—no vector database, no embeddings server. The core question: Does it actually deliver cheap, accurate, honest retrieval at scale, or does it break? We tested it the empirical way: find the boundary, report it.

What's In The Data vs. What's Asked

The corpus: 73 real registry field definitions (~2,562 tokens) spanning Cardiology registries.

Test 1: Basic Retrieval

Reasoning: Before adversarial testing, does BM25 surface the correct answer doc for a straightforward question? We ingested the 73 documents and asked 3 real questions to inspect which docs ContextForge places in the prompt (model-free).

Results: Top hit correct every time

  • "Which MCS device types?" → mcs_type ranked #1
  • "NCDR access sites + door-to-balloon?" → access_site #1, door_to_balloon #2
Nuance: It retrieves a slice (~20 docs), not a laser 1-2 docs. The top hit is right, but ~18 other loosely-related fields ride along on shared vocabulary. For 73 docs, the token win over stuffing everything is only ~2x (1,200 vs 2,562 tokens). The real win requires scale.

Test 3: The Retrieval Wall

Reasoning: Tests 1 & 2 were friendly, relying on exact keyword overlap. The honest question is: where does BM25-without-embeddings break? We designed three adversarial classes to find the wall.

A. Paraphrase Gap

0 shared content words results in a silent MISS at scale. When query vocabulary diverges from doc vocabulary, retrieval goes blind.

B. Aggregation

BM25 cannot compute max/min. The model sees 20 docs, not all 4,000. It is structurally impossible to answer superlatives confidently.

C. Absence / Abstention

Retrieval has no abstain path. It always returns the top 20 hits. Handing pure distractors to the model is the leading risk of hallucination.

ContextForge is great at "find the doc with these exact terms," but breaks on paraphrase, aggregation, and absence.

There, the trap contained a decoy: a plausible-but-wrong number sitting in the context, waiting to be misattributed. Here, ContextForge's retrieval miss handed the model clean distractors—no plausible wrong answer present—so abstaining is easy.

That raised a hypothesis worth testing directly: does the danger return if a plausible decoy value rides along in the retrieved slice? So we ran a dedicated decoy test.
Test 5 · The Decoy Test

We Tried To Break The Safety Net—On Purpose

Reasoning: Test 4 fed models clean distractors and they abstained perfectly. But the report's own hypothesis was that the real danger is a retrieval miss where a plausible wrong value co-occurs. That claim was an inference, not a measurement. So we tested it.

Method: We planted a decoy document in the corpus phrased almost identically to the question, differing in one noun. Ask "maximum permissible admission potassium?" and the retrieved slice now contains "maximum permissible admission lactate value the validator accepts is 22 mmol/L." A model that grabs 22 and attributes it to potassium has hallucinated by misattribution—the exact failure that broke small models in workingset. We verified ContextForge actually retrieved every decoy into the prompt (precondition met), then ran 3 decoy questions × 8 models × 3 reps = 72 trials with the same strict prompt.

Result: 0 / 72 hallucinations. The hypothesis did NOT reproduce.

Every model—including gpt-5.4-mini and gpt-5-mini, the two that cracked 33-67% in workingset—abstained on all 72 decoy trials. The most telling trial: claude-haiku-4.5 answered "Not in the provided data. The provided knowledge specifies a maximum of 6 simultaneous vasopressors, but does not contain [inotropes]." The model saw the decoy value, recognized it belonged to a different field, and explicitly refused to misattribute it. Only 1 of 72 answers even surfaced the decoy number—and that one flagged the mismatch.
The honest caveat: This refutes the danger-zone hypothesis under these conditions—but the conditions differ from workingset in two ways that matter. (1) Our decoy was cross-field (a lactate value vs. a potassium question), which any competent model can reject; workingset's decoy lived in the same semantic field as the question. (2) Our system prompt strictly licensed abstention ("if absent, reply exactly 'Not in the provided data'"); workingset used a softer framing. So we ran one more test to remove the second variable—a permissive prompt (Test 6, below). That's where the trap finally bit.
Test 6 · The Permissive-Prompt Control

Take Away The Strict Prompt, And The Trap Finally Bites

Test 4 and Test 5 both used a strict prompt: "Use ONLY the context. If the answer is absent, say so. Do NOT guess." That left a question we hadn't answered: is the model intrinsically safe, or is the strict prompt doing the work? So we re-ran the three trap questions (2 absence + 1 paraphrase-miss) across all 8 models with a naive, permissive prompt—"Answer the question using the knowledge provided. Be concise."—no abstention license, no anti-guess clause. The naive-developer prompt. 72 trials, only the system prompt changed.

Result: 5 / 72 hallucinations (~7%). The strict prompt WAS load-bearing.

Strict prompt: 0/72. Permissive prompt: 5/72. So the prompt is not cosmetic—removing it produced real hallucinations. But the pattern is the whole story: all 5 were the same question, and only the two mini models.

The mechanism—and it's not "small models are dumber": Every hallucination was the mini models answering "90 minutes" to "longest acceptable door-to-balloon time." 90 minutes is the famous real-world cardiology standard—it lives in every model's training data. It is NOT in our corpus (the corpus never states a max, and CF's paraphrase miss meant the door-to-balloon doc was never even retrieved). So this isn't random fabrication: it's training-data substitution. When retrieval whiffed and the prompt didn't forbid guessing, the small models filled the gap with a well-known fact from memory instead of staying grounded in the context.

The two absence questions (potassium, smoking) stayed at 0/72 even permissive—because there is no "registry potassium maximum" in any model's training data to substitute. Nothing to grab, so they abstain regardless of prompt. The trap requires BOTH a retrieval miss AND a famous answer the model already knows.
What this means in practice: The strict prompt isn't insurance against gibberish—it's specifically what stops a model from substituting a well-known training-data fact when CF's retrieval missed the real document. Two concrete rules fall out: (1) always pair ContextForge with the strict "use only the context, don't guess" prompt, and (2) don't route grounded retrieval through a mini model—the frontier models (opus / sonnet / gpt-5.5 / gpt-5.4) stayed grounded even under the permissive prompt; only the minis leaned on the prior.

The Bottom Line

  • Flatness claim: ROCK SOLID. 1,136x disk growth results in 1.00x prompt size.
  • Retrieval failures: 3 architectural failure modes (paraphrase, aggregation, absence)—all stemming from the lack of embeddings.
  • Safety is conditional, and the condition is the prompt. With a strict "use only the context, don't guess" prompt: 0/144 hallucinations (clean distractors + planted decoys). Strip that prompt: 5/72 (~7%)—all on one paraphrase-miss question, all from the two mini models, all "90 minutes" substituted from training data. The model is a safety net only when (a) the prompt forbids guessing and (b) you're not using a mini for grounded retrieval. The frontier models stayed grounded even permissive.
  • For Daily Work: Fast, zero-infra POC scaffold for "chat with exact-keyword data." Weak at whole-dataset reasoning. Pair with a frontier model and set expectations accordingly.

Methodology: Model-free retrieval tests + real-model trials via GitHub Copilot CLI (mechanical grading).
